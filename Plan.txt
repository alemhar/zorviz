# Project Plan: Zorviz - Modular Local-First Platform

## 1. Executive Summary

**Goal:** Build a **Strict Local-First Platform** that serves as the operating system for small businesses. The Cloud is an *optional* extension, not a requirement.
**Architecture:** **Modular Core**. The system consists of a "Kernel" (Auth, Sync, Settings) and pluggable "Business Modules" (Auto Repair, Retail, Restaurant).
**Initial Scope:** Module A: "Repair & Service" (Targeting Car/Gadget Shops).

## 2. Core Philosophy

### 2.1 Strict Local-First
*   **Zero Dependency:** The Desktop App must boot, run, and provide full utility forever without *ever* touching the internet.
*   **Sync is a Feature:** Syncing to the cloud is an add-on service for backup and multi-device access, but the local database is the source of truth for that device.

### 2.2 Modular "lego" Architecture
Instead of a monolithic app, we build a **Core Kernel** and **Feature Modules**.

*   **Core Kernel:**
    *   Authentication (Local & Remote)
    *   Replication Engine (Sync)
    *   User Management
    *   Hardware IO (Printers, Scanners)

*   **Module A: Service & Repair (The "Zorviz" MVP)**
    *   *Features:* Asset Tracking (VIN/IMEI), Repair Orders, Service History.
    *   *Target:* Talyer, Phone Repair, Watch Repair.

*   **Module B: Retail & Inventory (Future)**
    *   *Features:* POS, Barcode Scanning, Stock Management.
    *   *Target:* Sari-sari stores, minimarts.

*   **Module C: Scheduling (Future)**
    *   *Features:* Appointments, Calendar, Resource Allocation.

## 3. Technical Architecture

### 3.1 The "Local Kernel" (Desktop/Tablet)
*   **App:** Tauri (Rust + React).
*   **Database:** SQLite.
*   **Schema Strategy:**
    *   **Core Tables:** `users`, `sync_state`, `devices`.
    *   **Module Tables:** Loaded on demand/config. e.g., enabling "Repair Module" creates `assets` and `job_orders` tables.

### 3.2 The "Cloud Swarm" (SaaS Layer)
*   **Role:** An infinite storage and relay service. It does not "run" the business logic; it mirrors the state of local nodes.
*   **App:** Next.js (Management Dashboard).
*   **Database:** Postgres (Supabase/Neon).

### 3.3 Module Interface
To make features modular, we will structure the code directories by domain, not technology.

```
packages/
  core/           # Types, base utilities
  db/             # Schema definitions
  modules/
    repair/       # Schema, React Components, Logic for Repair
    inventory/    # Schema, React Components, Logic for Inventory
    crm/          # Schema, React Components, Logic for Customers
```

## 4. The Workflow (Revised)

### Node Types
1.  **Commander Node (PC):** The main server for the shop. Holds the full SQLite DB.
2.  **Scout Node (Mobile/Tablet):** Connects to Commander via LAN.
3.  **Cloud Node (Web):** Optional backup & remote viewer.

### Data Flow
1.  **Intake:** Mechanic enters data on Scout Node -> Pushed to Commander (LAN).
2.  **Processing:** Service Advisor finalizes Estimate on Commander (Local).
3.  **Backup (Optional):** Commander syncs encrypted diffs to Cloud Node (Internet).

## 6. Detailed Feature Flows

### 6.1 Repair Module Workflow
The "Universal Repair" flow works for Cars, Computers, or Watches.

> [!IMPORTANT]
> **Mobile First for Mechanics:** All Mechanic-facing views (Job Board, Execution) MUST be optimized for **Mobile Phones** (small screens, touch targets), not just Tablets.

0.  **Scheduling / Booking (Optional)**
    *   **Source:** Cloud (Customer) OR Local (Advisor).
    *   **Action:** Create a `Booking` linked to an Asset + Time.
    *   **Result:** Appears on "Today's Schedule". Converting a Booking -> Job Ticket skips Step 1.

1.  **Asset Recognition (Advisor PC & Mechanic Mobile)**
    *   **Action:** User types Plate #, VIN, or Serial #.
    *   **Logic:**
        *   *Found:* Show Asset History (Past 5 visits).
        *   *New:* Auto-create Asset with "Draft" owner. (Reduce friction).

2.  **Intake & Triage (Advisor PC & Mechanic Mobile)**
    *   **Action:** Create a "Job Ticket" (from Scratch or Booking).
    *   **Data:**
        *   "Customer Complaint" (e.g., "Aircon not cold").
        *   "Initial Inspection" (Checklist).
        *   "Photos" (Upload to Local File Store).
    *   *Result:* Ticket Status = `TRIAGE`.

3.  **Estimation (Advisor PC)**
    *   **Action:** Advisor converts Ticket to Estimate.
    *   **Data:** Add Service Lines (Labor) and Parts (Inventory).
    *   *Result:* Ticket Status = `ESTIMATE`.

4.  **Approval (Hybrid)**
    *   **Action:** Customer approves via OTP, Signature (Local), or Web Link (Cloud).
    *   *Result:* Ticket Status = `APPROVED`. Deduct Inventory allocation.

5.  **Execution (Mechanic Mobile)**
    *   **Action:** Mechanic views "My Jobs" (Mobile Card View). Checks off line items.
    *   **Result:* Ticket Status = `WORK_IN_PROGRESS` -> `DONE`.

6.  **Completion & Billing (Advisor PC)**
    *   **Action:** Cashier generates Invoice.
    *   *Result:* Ticket Status = `PAID`.
